# Chinese Checkers / 跳棋游戏

一个基于 **C++17、Visual Studio 2022 和 EasyX 图形库** 编写的图形化跳棋游戏。项目使用标准 121 孔星形棋盘，支持 2–6 名玩家、每人 3、6 或 10 个棋子，并实现了鼠标与键盘操作、一步移动、BFS 连续跳跃、落点二次确认、每步倒计时、单方总时长、超时判负、完整排名、局势分析、机器人玩家、返回上一步和 UTF-8 日志记录等功能。

**v4.10** 在 v3.7 的计时与局势分析基础上，正式加入机器人玩家系统，并进一步完善最终区域规则、超时淘汰和比赛排名。玩家可以设置 0 至当前玩家总数的机器人，支持纯真人、真人与机器人混合以及全机器人自动对局。机器人会在五秒内分阶段展示选棋、选落点和执行移动，并通过推进评分、终点区撤离、重复局面检测和防循环策略选择走法。

## 2. 项目简介

本项目实现了一个本地运行的图形化跳棋游戏。程序启动后，玩家可以在统一设置界面中选择：

- 每步允许时间；
- 每名玩家单方总时长；
- 参加游戏的玩家人数；
- 每名玩家的棋子数量；
- 机器人玩家数量。

进入游戏后，系统按照玩家编号从小到大轮流行动。真人玩家可以通过鼠标点击棋子和落点，也可以使用键盘数字编号完成选棋与落子。机器人玩家则由程序自动分析当前棋盘，在五秒内依次展示“选中棋子—选择落点—确认移动”的过程。

棋盘采用标准跳棋星形结构，共 17 行、121 个孔位。每名玩家从自己的起始三角区出发，目标是将全部棋子移动到正对面的最终区域。棋盘孔位和移动关系并非手工逐个填写，而是通过几何坐标和距离关系自动生成相邻边与跳跃边。连续跳跃使用 BFS 搜索一次性计算所有合法落点。

v4.10 还增加了完整比赛排名。玩家全部棋子进入自己的最终区域后，会获得当前名次并退出后续行动；其他玩家继续比赛。超时或违反最终区保护规则的玩家会被判负，其棋子从棋盘清除，并根据判负时的完成数量和判负先后确定末尾名次。当只剩最后一名有效玩家时，系统自动补齐剩余排名并结束游戏。

为了减少机器人来回移动、重复布局和残局停滞，机器人模块会记录整盘局面、本方棋子布局、重复移动边和连续无推进回合，并根据剩余时间动态提高推进紧迫度。项目还提供 `tests/robot_simulation.cpp`，用于批量模拟六机器人十子对局并检查循环、终点区撤离和回合上限等问题。

整体代码拆分为棋盘、游戏流程、绘图、键盘、计时、局势分析、机器人、日志和数据结构等模块，便于阅读、调试、课程答辩和后续扩展。

## 3. 已实现功能

### 3.1 游戏设置

- 支持选择 2、3、4、5、6 名玩家。
- 支持每名玩家使用 3、6 或 10 个棋子。
- 支持设置 0 至当前玩家总数的机器人。
- 机器人优先分配给编号靠后的玩家。
- 支持纯真人对局。
- 支持真人与机器人混合对局。
- 支持全机器人自动对局。
- 支持选择每步时长：15 秒、30 秒、60 秒、90 秒。
- 支持选择单方总时长：5 分钟、10 分钟、15 分钟、30 分钟。
- 默认设置为 3 人、每人 3 个棋子、0 个机器人、每步 30 秒、单方 5 分钟。
- 设置界面按五行显示：
  - 每步时长；
  - 单方总时长；
  - 游戏人数；
  - 棋子数量；
  - 机器人数量。
- 支持鼠标点击 `-`、当前值和 `+` 按钮调整。
- 支持方向键操作：
  - `↑ / ↓` 切换设置项；
  - `← / →` 调整当前值；
  - `Enter` 按当前设置开始游戏。
- 玩家人数降低时，机器人数量会自动限制在玩家人数以内。
- 根据玩家人数自动分配对称营地。
- 每名玩家的目标区域为起始营地正对面的三角区。
- 支持红、蓝、绿、橙、紫、青六种棋子颜色。

### 3.2 棋盘系统

- 使用标准跳棋星形棋盘。
- 棋盘总行数为 17 行。
- 每行孔位数量为：

```text
1, 2, 3, 4, 13, 12, 11, 10, 9, 10, 11, 12, 13, 4, 3, 2, 1
```

- 棋盘总孔位数为 121。
- 自动生成每个孔位的屏幕坐标。
- 自动生成相邻移动边。
- 自动生成连续跳跃边。
- 自动识别鼠标点击到的孔位。
- 支持 2 人模式下的上下对称营地。
- 支持 3–6 人模式下的均匀营地分配。
- 三角区孔位按“从角尖到出口”的层次重新排序，为最终区规则和机器人撤离判断提供依据。

### 3.3 棋子移动规则

当前版本支持两类移动：

1. **一步移动**  
   棋子可以移动到相邻的空孔位。

2. **跳跃移动**  
   如果指定方向的相邻孔位存在棋子，并且该棋子后方对称位置为空，则可以跳到该空位。

一次行动可以进行连续跳跃。程序会直接搜索所有能够最终到达的孔位，玩家不需要逐段点击中间跳跃过程。

另外，v4.10 增加以下最终区域限制：

- 棋子进入自己的最终区域后，只能继续在自己的最终区域内移动，不能再次离开。
- 外部棋子不能主动进入其他玩家的最终区域。
- 如果棋子原本已经位于其他玩家最终区域，只允许向更靠近出口的外层移动。
- 连续跳跃可以经过其他玩家最终区域，但只有满足规则的孔位才能作为最终落点。
- 已超时、已判负或已完成玩家的棋子不再参与正常移动和占位判断。

### 3.4 连续跳跃搜索

- 使用队列执行 BFS 搜索。
- 使用 `visited` 集合记录已经到达的孔位。
- 避免连续跳跃过程形成无限循环。
- 使用虚拟局面模拟棋子在搜索过程中的当前位置。
- 起始孔位在虚拟局面中视为空。
- 当前模拟孔位在虚拟局面中视为有棋子。
- 其他棋子按照真实棋盘判断。
- 连续跳跃过程同时检查：
  - 中间孔位是否有棋子；
  - 目标孔位是否为空；
  - 是否违反自己的最终区只进不出规则；
  - 是否违反其他玩家最终区落点限制。
- 所有合法落点以绿色圆圈显示。
- 前 9 个合法落点显示键盘编号。

### 3.5 机器人玩家

- 新增独立的 `robot.h / robot.cpp` 模块。
- 机器人可以自动完成完整回合。
- 每个机器人回合严格按照五秒流程展示：
  - 约 2 秒后选中棋子；
  - 约 3.5 秒后显示目标落点；
  - 约 5 秒时确认并执行移动。
- 如果没有合法走法，机器人等待五秒后自动跳过。
- 机器人回合中，真人不能通过鼠标或键盘插入操作。
- 右侧玩家信息会标记“机器人”或“人工”。
- 机器人使用与真人完全相同的合法落点函数和移动执行函数。
- 机器人不直接改写棋盘，只返回棋子下标和目标孔位。
- 支持所有玩家均为机器人，用于自动演示或测试。

机器人评分会综合考虑：

- 棋子沿起点到终点方向的推进比例；
- 进入自己的最终区域；
- 在最终区域内向深处整理；
- 离开自己的起始区域；
- 回到起始区域的惩罚；
- 一步或连续跳跃的有效距离；
- 当前剩余总时间；
- 连续没有推进的回合数；
- 每名机器人随机生成的轻微推进风格差异；
- 是否立即撤销同一棋子的上一步；
- 整盘局面是否重复；
- 本方棋子布局是否重复；
- 同一枚棋子是否反复走同一条边；
- 是否停留在其他玩家的最终区域；
- 是否能够及时离开高风险最终区域；
- 残局时是否优先推动尚未入营的棋子。

### 3.6 反循环与随机策略

机器人模块维护以下跨回合数据：

- 整盘棋局哈希出现次数；
- 每名机器人自身棋子布局哈希出现次数；
- 每枚棋子重复走同一条边的次数；
- 每名机器人连续无有效推进的回合数；
- 每名机器人本局的轻微推进风格系数。

机器人会：

- 大幅惩罚立即反向撤回；
- 大幅惩罚已经出现过的完整局面；
- 惩罚重复的本方布局；
- 惩罚同一棋子的重复边；
- 连续六回合无推进后，只要存在前进走法，就过滤横移和后退；
- 本方布局第二次重复后，强制保留推进量最大的走法；
- 在分数接近最优值的高质量候选中进行加权随机选择；
- 当重复情况严重时，将候选分差缩小为 0，只选择当前最高分走法。

这样可以在减少循环的同时，让相同设置下的不同对局保留一定变化。

### 3.7 最终区域保护规则

v4.10 增加了最终区域保护机制：

- 最终区域保护阈值为该玩家棋子总数的 60%，向上取整。
- 3 个棋子时，达到 2 个进入最终区后触发保护。
- 6 个棋子时，达到 4 个进入最终区后触发保护。
- 10 个棋子时，达到 6 个进入最终区后触发保护。
- 当区域主人在某一步首次达到保护阈值时，程序检查该最终区域内是否仍有其他玩家棋子。
- 仍未离开的外来棋子所属玩家会被判定为“终点区未及时离开判负”。
- 同一玩家有多枚棋子滞留时，只判负一次。
- 被判负玩家的棋子会从棋盘清除。
- 判负时会记录其已经完成的棋子数量和判负顺序。

机器人会提前按照 30% 警戒线规划撤离：

- 当某个区域主人接近或达到 30% 完成进度时，机器人会提高该区域的风险等级。
- 如果机器人有棋子滞留在对方最终区，会优先选择撤离走法。
- 优先撤离靠近角尖、未来更容易被封死的棋子。
- 优先减少剩余撤离层数。
- 优先保留更多可用出口。
- 当区域主人达到 30% 且仍有外来棋子撤离时，区域主人控制的机器人会尽量避免继续封住出口。

30% 仅用于机器人提前规划；真正触发玩家判负的规则阈值仍为 60%。

### 3.8 回合、排名与判负

- 行动顺序固定为玩家编号从小到大。
- 每次成功移动后自动切换到下一名有效玩家。
- 已完成或已判负玩家从行动顺序中移除。
- 某名玩家全部棋子进入自己的最终区域后：
  - 标记为已完成；
  - 按完成顺序获得名次；
  - 从后续行动顺序中移除；
  - 其他玩家继续比赛。
- 第一名玩家保存在 `winnerindex` 中。
- 当只剩一名有效玩家时，该玩家自动获得剩余名次。
- 每步时间或单方总时长耗尽时，当前玩家超时判负。
- 违反最终区保护规则的玩家也会判负。
- 判负玩家的末尾排名按照以下顺序实时计算：
  1. 判负时完成棋子数量更多者靠前；
  2. 完成数量相同时，后判负者靠前；
  3. 先判负者排名更靠后。
- 游戏结束界面显示全部玩家最终排名。
- 判负玩家显示判负原因和完成进度。
- 正常完成玩家显示完成名次。
- 最近完成的玩家会在信息栏中突出提示。

### 3.9 计时与局势分析

- 支持每步倒计时。
- 支持每名玩家单方总时长。
- 当前回合同时扣除本步时长和单方总时长。
- 机器人五秒行动过程同样计时。
- 超时后立即调用统一淘汰流程。
- 玩家完成或判负后停止其计时。
- 右侧显示当前步剩余时间。
- 右侧显示每名玩家总剩余时间。
- 局势分析在开始游戏、每次移动、判负和撤销后更新。
- 局势评分考虑：
  - 棋子推进程度；
  - 跨越中线后的推进奖励；
  - 己方棋子是否能形成配合跳跃；
  - 已进入最终区域的棋子数量；
  - 是否给其他玩家提供跳板。
- 所有有效玩家评分折算成总和为 100% 的概率。
- 已判负玩家评分和概率为 0。
- 局势百分比属于启发式展示，不是严格数学胜率。

### 3.10 图形界面功能

- 棋子使用六种颜色区分玩家。

- 当前玩家目标区域使用浅绿色高亮。
- 合法落点使用绿色圆圈标记。
- 已预览但尚未确认的落点使用蓝色标记。
- 玩家棋子显示键盘编号。
- 合法落点显示键盘编号。
- 右侧显示当前棋子编号或落点编号列表。
- 右侧显示每名玩家：
  - 人工或机器人身份；
  - 已完成棋子数量；

### 3.11 落点二次确认

v4.10 将落点操作改为二次确认：

```text
选择棋子
   ↓
显示全部合法落点
   ↓
第一次点击或按数字键选择落点
   ↓
落点变为蓝色预览状态
   ↓
再次点击或再次按相同数字
   ↓
执行真实移动
```

- 第一次选择落点不会立即移动。
- 选择其他绿色落点可以更改预览位置。
- 点击“返回上一步”或按返回键时，会先取消待确认落点。
- 再次返回才会取消棋子选择。
- 机器人同样按照“选棋—选落点—执行移动”分阶段展示。

### 3.12 返回与重新开始

- 设置阶段可以继续修改五项设置。
- 选择落点阶段第一次返回时取消预览落点。
- 没有预览落点时返回会取消当前棋子选择。
- 游戏进行阶段可以撤销最近一次真实移动。
- 撤销后恢复棋子位置和回合顺序。
- 撤销后重新启动当前玩家计时。
- 撤销后重新计算局势分析。
- 撤销后重置机器人状态。
- 支持 `Esc` 返回上一步。
- 支持 `Delete` 返回上一步。
- 选择落点阶段按数字 `0` 返回。
- 游戏结束后支持重新开始。
- 一旦已经产生正常排名或判负记录，不能继续撤销更早走法，避免破坏排名和淘汰状态。
- 判负处理会清空历史记录，防止撤销到已经无效的玩家回合。

### 3.13 日志系统

- 日志使用 UTF-8 with BOM 编码。
- 日志记录：
  - 程序启动；
  - 每步时长和总时长调整；
  - 人数调整；
  - 棋子数量调整；
  - 机器人数量调整；
  - 玩家营地与机器人身份；
  - 固定行动顺序；
  - 真人选棋和落点预览；
  - 机器人选棋、选落点和移动；
  - 真实棋子移动；
  - 超时判负；
  - 最终区未及时离开判负；

### 3.14 自动模拟测试

项目提供：

```text
ChineseCheckers/tests/robot_simulation.cpp
```

该测试程序用于运行不等待界面动画的全机器人模拟。测试设计包括：

- 构建六机器人、每人十子的完整对局。
- 单盘最多执行 2000 次移动。
- 连续运行 60 盘。
- 验证 3、6、10 子对应的 60% 最终区保护阈值。
- 验证外部棋子不能进入其他玩家最终区域。
- 验证原本位于其他玩家最终区的棋子仍能向出口撤离。
- 验证完整对局能够在限制内结束。
- 检查同一整盘局面的最大重复次数不超过 2。
- 检查单名玩家使用回合数不超过 60。
- 检查机器人不会在 30% 警戒后错过合法撤离机会。
- 检查机器人不会因策略失误触发最终区滞留判负。
- 输出各盘移动次数、重复次数、最大回合数、自然完成人数和胜者分布。

测试源文件为独立控制台测试入口，不属于主 EasyX 游戏工程。发布或修改机器人算法后，建议重新编译并运行该测试。

## 8. 主要算法与实现思路

### 8.1 游戏状态机

项目继续使用：

```cpp
enum phasekind
{
    phase_player_count,
    phase_piece_count,
    phase_select_piece,
    phase_select_target,
    phase_game_over
};
```

新版设置主要集中在 `phase_player_count`，但保留旧阶段以兼容原流程。

### 8.2 标准星形棋盘生成

程序根据每行孔位数量自动计算坐标，不需要手工保存 121 个坐标。三角区域会从真实角尖向出口重新排序，为最终区层数判断提供稳定顺序。

### 8.3 相邻边与跳跃边生成

- 距离约等于一个孔位间距：建立相邻边。
- 距离约等于两个孔位间距，且中点存在孔位：建立跳跃边。
- 通过几何关系自动生成六个方向上的移动关系。

### 8.4 BFS 连续跳跃

```text
起点进入队列
      ↓
取出当前模拟位置
      ↓
枚举全部跳跃边
      ↓
中间有棋子且目标为空
      ↓
检查最终区规则
      ↓
目标未访问则加入队列
      ↓
满足落点限制则加入最终结果
```

搜索过程允许穿过其他玩家最终区，但不会把非法孔位加入最终落点。

### 8.5 最终区层数规则

最终区孔位从角尖开始按层排列：

```text
第 0 层：角尖
第 1 层：2 个孔位
第 2 层：3 个孔位
……
```

外来棋子原本位于该区域时，只能从较小层数走向较大层数，也就是向出口撤离，不能继续进入更深层。

### 8.6 完整排名算法

正常完成玩家按顺序写入：

```cpp
std::vector<int> ranking;
```

玩家完成后：

```text
finished = true
rank = ranking.size() + 1
加入 ranking
从 turnorder 移除
```

判负玩家单独按照完成数量和判负先后计算末尾排名。最后剩余玩家自动获得尚未分配的最前名次。

### 8.7 统一淘汰流程

超时和最终区违规共用 `game_eliminate_players()`：

1. 停止计时；
2. 记录完成进度；
3. 记录判负顺序和原因；
4. 清除该玩家全部棋子；
5. 从行动顺序移除；
6. 清空历史；
7. 重置机器人；
8. 更新判负排名；
9. 更新局势分析；
10. 判断是否只剩最后玩家；
11. 未结束则启动下一回合。

### 8.8 机器人推进评分

机器人通过起始区中心和目标区中心构造推进轴，将棋子位置投影到该轴上，得到推进比例。走法评分的基础部分为：

```text
推进增量 × 推进权重 × 时间紧迫度 × 机器人风格
```

时间越少、连续无推进回合越多，前进权重越高。

### 8.9 机器人最终区撤离

机器人为对方最终区设置 0–3 级风险：

- 0：不在对方最终区；
- 1：普通滞留；
- 2：区域主人接近 30%；
- 3：区域主人已达到 30%。

撤离评价重点考虑：

- 是否存在无路可走的滞留棋子；
- 全部滞留棋子距离出口还剩多少层；
- 当前可用撤离落点数量；
- 是否能直接离开区域；
- 是否优先移动最靠近角尖的危险棋子。

当撤离任务激活时，普通推进和配合评分不能覆盖撤离要求。

### 8.10 机器人重复局面检测

机器人使用 FNV 风格哈希记录：

- 完整棋盘局面；
- 单个玩家全部棋子排序后的布局；
- 单枚棋子的有向移动边。

通过布局排序，本方同色棋子编号互换不会被误判为不同布局。

### 8.11 机器人候选过滤

候选生成后依次过滤：

- 紧急撤离时只保留撤离价值最高的走法；
- 提前撤离规划时保留更安全路线；
- 区域主人有外来棋子撤离时避免继续封住出口；
- 残局优先移动尚未进入自己最终区的棋子；
- 有非反向走法时删除立即撤回；
- 有未出现局面时删除重复局面；
- 有未重复本方布局时删除本方重复布局；
- 长时间无推进时删除横移和后退。

### 8.12 高质量随机选择

过滤后找到最高分 `bestscore`，通常保留与最高分相差不超过 650 的候选，再按照分数差进行加权随机。

严重重复时允许分差变为 0，只选择最高分，避免随机继续形成循环。

### 8.13 五秒机器人流程

```text
0–2 秒：思考并计算最优走法
2 秒：显示选中棋子
3.5 秒：显示目标落点
5 秒：执行真实移动
```

如果无合法走法，五秒后自动跳过。

### 8.14 落点二次确认

真人和机器人都先设置：

```cpp
robotpreviewtarget
```

只有再次确认同一落点时，才调用真实移动函数。这样可以降低误操作，并让自动走棋过程更容易观察。

### 8.15 计时算法

使用 `GetTickCount()` 获取真实毫秒数，每次累计满 1000 毫秒后扣减整数秒。机器人等待过程也属于真实回合时间。

### 8.16 局势概率折算

先计算所有有效玩家分数，再按比例折算为整数百分比。最后一个有分玩家接收四舍五入余数，确保概率总和为 100%。

### 8.17 自动模拟测试逻辑

测试程序使用不等待动画的方式连续调用 `robot_make_decision()`，并检查：

```text
goalrules == true
ended == true
maxrepeat <= 2
maxplayerturns <= 60
zoneviolations == 0
evacuationmisses == 0
```

测试源代码定义了验收标准；是否通过仍应以本地重新编译和实际运行结果为准。

- 局势分析；
- 返回上一步；
- 重新开始；
- UTF-8 BOM 日志；

## 11. 未完成内容与后续可优化方向

1. **增加机器人难度等级**  
   当前机器人统一使用同一套评分策略。

2. **增加逐个玩家类型设置**  
   当前机器人固定分配给编号靠后的玩家。

3. **增加更深层局面搜索**  
   当前机器人属于单步启发式决策，没有多回合博弈搜索。

4. **优化多人对局策略**  
   当前主要关注自身推进、撤离和循环控制，对其他玩家整体布局预测有限。

5. **完善标准规则配置**  
   最终区 60% 保护属于本项目自定义规则，可改为可选规则。

6. **增加存档与读档**  
   当前只能记录日志，不能恢复关闭前的对局。

7. **增加移动与跳跃动画**  
   当前棋子移动仍然是瞬间完成。

8. **增加音效与机器人状态动画**  
   可以增加倒计时、选棋、判负和完成提示。

9. **完善排名规则配置**  
   可以增加剩余时间、总推进度等排名因素。

10. **补充真人操作自动化测试**  
    当前专项测试主要覆盖机器人和最终区规则。

11. **将测试项目加入解决方案**  
    当前测试源文件为独立控制台入口，没有作为第二工程正式加入解决方案。

12. **提高跨平台能力**  
    当前依赖 EasyX 和 Windows API。

## 16. 结语

v4.10 的重点并不只是增加“电脑自动走棋”，而是将机器人正式接入原有棋盘、计时、排名和输入系统。真人和机器人最终都调用同一套合法落点和移动函数，从而避免出现两套规则。

新版最终区域保护机制解决了外来棋子长期占据停车区、阻止区域主人完成的问题；30% 预警和 60% 判负阈值则使机器人能够提前撤离，而不是等到规则触发后才处理。完整排名系统也让游戏从“产生第一名后结束”扩展为“所有玩家名次逐步确定”。

机器人通过推进评分、残局处理、撤离优先、重复局面哈希、反向移动惩罚、重复边统计和高质量随机选择，降低了来回循环和长时间停滞。独立模拟测试源文件进一步为机器人和规则修改提供了回归检查基础。

通过本项目，可以练习 C++ 模块化设计、EasyX 图形绘制、图结构建模、BFS 搜索、状态机控制、实时计时、启发式决策、哈希去重、多人排名和自动化模拟测试。希望本项目能为 C++ 课程设计、跳棋程序、图形化小游戏和基础游戏 AI 开发提供参考。
